# הקשר מערכת — My Wall CRM

מסמך חי. מעדכנים אותו כשיש שינוי משמעותי במטרות, במה שעובד, או בהחלטות חשובות.
הסוכן אמור לקרוא אותו בתחילת כל שיחה.

---

## מה אנחנו בונים

מערכת ניהול תפעולית לקיר הטיפוס **My Wall**.
המערכת מנהלת לידים ולקוחות, חוגים, נוכחות, הצהרות בריאות, תשלומים, הודעות WhatsApp ותפעול יומיומי.

שני תפקידים עיקריים:
- **מנהל** — גישה מלאה
- **צוות** — לידים, חוגים, נוכחות, מסוף כניסה, הצהרות

---

## מטרות

1. לרכז במקום אחד את כל התפעול של הקיר
2. להקטין עבודה ידנית (נוכחות, דיוור, תשלומים, קליטת לקוחות)
3. לחבר WhatsApp בצורה יציבה וישירה מול Meta
4. לשמור נתונים בצורה מאובטחת ב-Supabase
5. לאפשר לצוות לעבוד מהר ביום-יום בלי להסתבך

---

## מה כבר עובד

- דשבורד כללי
- מסוף כניסה מהיר
- לקוחות ולידים (הורים + מתאמנים, תיק לקוח, שיחות)
- לוח חוגים ונוכחות (כולל יצירה יומית אוטומטית)
- תצוגת נוכחות לילד בתיק לקוח: לוח חודשי עם תגי סטטוס (ממתין למילוי / הגיע / לא הגיע / חליפי / יום חג)
- דיוור WhatsApp: שידורים, קהלים, תבניות, תשובות שמורות
- קופה וחשבוניות דרך iCount
  - קישור סליקה בוואטסאפ מפיק חשבונית מס קבלה אוטומטית אחרי תשלום (דרך עמוד הסליקה + הודעת שרת חזרה אלינו)
  - מסך מכירה בדלפק: מנויים, כרטיסיות, מוצרים; חשבונית מס קבלה / הצעת מחיר; ניקוב כרטיסייה בתיק לקוח ובמסוף כניסה
- מחירון
- בטיחות: בדיקות יומיות ודיווח אירועים
- עובדים ומשמרות
- מבחנים (רמה / אבטחה / הובלה)
- הצהרות וטפסים + מעקב חתימות + PDF לתיק לקוח
- אוטומציות / מסעות לקוח
- טפסים ציבוריים: הצטרפות, הצהרת בריאות, קליטה, פרטיות
- בוט תשובות אוטומטיות (Gemini) עם לוח שעות

---

## מה בתהליך / פערים ידועים

- מעבר מ-Wassenger ל-Meta ישיר — לא לנתק את הישן לפני שכל הבדיקות עוברות
- חיזוק אבטחה והחלפת מפתחות שנחשפו בעבר
- חלק מהנתונים בטבלאות ייעודיות וחלק באוסף כללי (`kv_collections`)
- Instagram / Messenger קיימים ברמה בסיסית, פחות בשלים מ-WhatsApp
- אונבורדינג מסמכים: תעודות זהות וקבצי PDF לתיק לקוח
- תבניות הודעות מאושרות חייבות להיות במסד החי (לא רק במטמון מקומי) — אחרת חלונית התקשורת נשארת בלי אפשרויות בחירה
- כפילות כרטיסי הורה לאותו טלפון בפורמטים שונים (למשל `050…` ו-`972…`) — תוקן: חלון 24ש נגזר גם מהודעות בשרשור וגם מאחים, והודעה נכנסת מעדכנת את כל הכרטיסים התואמים. עדיין כדאי לאחד כרטיסים כפולים בעתיד

## מבנה טכני בקצרה

| שכבה | מה משתמשים |
|---|---|
| אתר | React + Vite |
| שרת | Node + Express |
| כניסה למשתמשים | Supabase Auth |
| מסד נתונים | Supabase (Postgres) |
| מטמון מקומי | `server/db.json` (לא מקור האמת) |
| פריסת API | Render |
| פריסת אתר | Vercel |

פורטים מקומיים:
- אתר: `3000`
- שרת: `5000`

תיקיות עיקריות:
- `client/` — ממשק המשתמש
- `server/` — API ולוגיקה
- `database/` — מיגרציות SQL

## החלטות חשובות

- Supabase הוא מקור האמת לנתונים
- `db.json` הוא מטמון מקומי בלבד
- WhatsApp עובר לחיבור ישיר מול Meta
- לא לנתק Wassenger מוקדם מדי
- תשובות למשתמש תמיד בעברית פשוטה בלבד — בלי ערבוב אנגלית באותה שורה, בלי תעתיק כמו „פרודקשן”, ובלי ראשי תיבות כמו PDF בתוך משפט עברי (ראו כלל העברית)
